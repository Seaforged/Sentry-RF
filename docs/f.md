Findings first: I would not sign off on this tag as field-ready.

CRITICAL

Warmup poisoning is still live. C:\Projects\sentry-rf\Sentry-RF-main\src\cad_scanner.cpp:1228-1255, 1272-1276, 1600-1627, and 1640-1644 still learn active taps as ambient during boot and still auto-tag anything first seen before AMBIENT_WARMUP_MS. A transmitter present at boot can still become invisible for the rest of the session. Exact fix: delete the recordAmbientTap(..., true) warmup learning paths and the firstSeenMs < AMBIENT_WARMUP_MS retag path, then replace them with a probation table that only graduates a frequency after the full window and only if it never picks up FHSS/RID/GNSS corroboration.
HIGH

Invalid WiFi RID still raises a real warning. C:\Projects\sentry-rf\Sentry-RF-main\src\wifi_scanner.cpp:375-399 and 416-436 set THREAT_WARNING and remoteIdDetected=true for any vendor IE with OUI FA:0B:BC, even when decodeBeaconRID() fails. A crafted beacon/action frame can force false RID alerts; a padded frame can also evade RID entirely because the callback truncates the body at MAX_PAYLOAD in C:\Projects\sentry-rf\Sentry-RF-main\src\wifi_scanner.cpp:97-103. Exact fix: only promote to WARNING and set remoteIdDetected after full ASTM validation succeeds; treat undecoded OUI hits as diagnostic-only or at most ADVISORY.
The serial/ZMQ race is not actually fixed. C:\Projects\sentry-rf\Sentry-RF-main\src\data_logger.cpp:63-103 still prints [ZMQ] even if xSemaphoreTake(serialMutex, 20ms) fails, while C:\Projects\sentry-rf\Sentry-RF-main\include\data_logger.h:26-30 and C:\Projects\sentry-rf\Sentry-RF-main\docs\SENTRY-RF_Known_Issues_Tracker.md:266-270 claim full serialization. Under contention, the bridge line can still interleave and corrupt JSON. Exact fix: remove the unlocked fallback and wrap both [ZMQ] Serial.printf calls in blocking SERIAL_SAFE(...), or take serialMutex with portMAX_DELAY before every emit.
WiFi frame loss under load is now proven, not hypothetical. C:\Projects\sentry-rf\Sentry-RF-main\src\wifi_scanner.cpp:75-112, 355-367, and 369-440 still enqueue every management frame, drain one frame per loop, and already log overflow. Real RID frames will be dropped in busy RF or during boot warmup. Exact fix: in the callback, reject non-beacon/non-action subtypes before enqueue, then in wifiScanTask drain the queue in batches until empty instead of one frame per pass.
NAN/action-frame RID is still not supported. C:\Projects\sentry-rf\Sentry-RF-main\src\wifi_scanner.cpp:186-215 walks 0xD0 action frames as if they were beacon IE lists. Exact fix: preserve the full 802.11 management frame and send action frames through a NAN-specific parser such as odid_wifi_receive_message_pack_nan_action_frame() instead of findRemoteIdIE().
The 2.4 GHz RF path is still blind to WiFi-channel control links. C:\Projects\sentry-rf\Sentry-RF-main\src\detection_engine.cpp:707-728 drops peaks inside WiFi channels outright, and C:\Projects\sentry-rf\Sentry-RF-main\src\cad_scanner.cpp:1078-1145 only scans LoRa-like 2.4 GHz channels. OcuSync-style links can still sit inside normal WiFi channels and disappear from RF detection. Exact fix: remove the isWiFiChannel(freq) hard reject and add a separate wideband plateau detector for contiguous elevated bins on 2.4 GHz.
GNSS integrity is still only half wired into operations. C:\Projects\sentry-rf\Sentry-RF-main\src\detection_engine.cpp:1342-1392 still ignores GNSS unless RF was recently active and exactly one candidate exists, DET_SOURCE_GNSS is defined but never emitted anywhere (C:\Projects\sentry-rf\Sentry-RF-main\include\detection_types.h:104-108), and C:\Projects\sentry-rf\Sentry-RF-main\src\gps_manager.cpp:210 still uses cno > 0 instead of cno >= GPS_MIN_CNO. C:\Projects\sentry-rf\Sentry-RF-main\src\detection_engine.cpp:300-311 still merges any hit anywhere in 860-870 or 902-928 into a candidate as soon as fhssSub is live.
